# Gamelytics 游戏运营数据看板

这是一个面向游戏运营岗位作品集的轻量级数据看板项目。项目刻意保留了原有的 HTML / CSS / Vanilla JS 结构与视觉风格，重点放在数据口径、离线聚合和运营分析表达，而不是做一套“看起来很大”的后台系统。

## 项目定位

- 用途：大陆游戏行业求职
- 目标岗位：偏数据分析的游戏运营

## 数据来源

原始数据位于 `data/raw/Gamelytics/`：

- `reg_data.csv`：注册数据，100 万用户
- `auth_data.csv`：登录数据，960 万+ 行
- `ab_test.csv`：A/B 变现实验样本，404,770 用户

由于 `auth_data.csv` 体量较大，页面不直接读取原始 CSV，而是先离线聚合，再让前端读取轻量结果文件。

当前仓库包含两类离线产物：

- 当前页面直接消费的聚合结果：`data/dashboard-metrics.json`
- 便于核对口径的预处理结果：`data/processed/`

## 指标口径

- DAU：按自然日去重登录用户数
- 留存：定点留存（exact-day retention），不是滚动留存
- Cohort：按注册月份聚合
- 变现与 A/B：仅代表 `ab_test.csv` 实验样本，不代表全部注册用户

这意味着：

- 全量活跃/留存分析与变现实验分析不是同一用户母体
- 因此不能把全量留存和实验样本 ARPU 直接理解为一个完整漏斗

## 留存解释边界

因为这里使用的是定点留存，所以留存曲线不一定单调下降，`D7` 高于 `D1` 并不自动说明计算错误。

对于最近注册的用户，部分留存窗口可能尚未成熟。项目在离线处理时会按“可观测用户”计算分母，而不是把未成熟窗口强行算成完整数据。

## A/B 测试解释边界

本项目保留了 A/B 模块，但定位是“实验样本结果解读”，不是“给出最终产品决策”。

当前 A/B 模块重点展示：

- 样本内付费率差异
- 样本内 ARPU 差异
- 基础统计显著性结果

因此页面中的 A/B 文案更适合回答：

- 实验样本里有没有明显优劣趋势
- 哪个指标可能需要进一步验证

而不是直接宣称某个方案已经可以全量上线。

## 项目结构

- `index.html`：页面结构与静态容器
- `gamelytics-dashboard.css`：现有视觉样式
- `gamelytics-dashboard.js`：读取聚合 JSON 并渲染页面
- `scripts/prepare_dashboard_data.py`：唯一的离线预处理脚本，同时生成页面使用的聚合结果与审计输出
- `data/dashboard-metrics.json`：当前页面直接读取的聚合结果
- `data/processed/`：按模块拆分的预处理输出

## 本地使用

生成页面聚合与预处理审计输出：

```bash
python3 scripts/prepare_dashboard_data.py
```

启动本地静态服务：

```bash
python3 -m http.server 8000
```

浏览器打开：

```text
http://127.0.0.1:8000/
```

说明：

- 页面通过 `fetch()` 读取 JSON，不建议直接双击打开 `index.html`
- 预处理脚本依赖 `pandas`

## 为什么这仍然适合作品集

- 没有为了“像企业后台”而重做一套 UI，重点仍然放在数据表达
- 离线聚合链路清晰，能说明你知道大体量日志不该直接丢进浏览器
- 指标口径明确，尤其区分了“全量活跃/留存”和“实验样本变现”
- A/B 结论没有强行美化，保留了负向结果和不显著结果，更接近真实业务分析
- 页面里的洞察被刻意写成“运营假设 / 风险信号”，而不是把相关性直接包装成因果定论
